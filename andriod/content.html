Intent理解显示与隐式区别

  • 显示Intent（Explicit Intent）是明确指定了目标组件的Intent。它通过指定目标组件的类名或包名与类名的方式来明确指定要启动的组件
  • 隐式Intent（Implicit Intent）是没有明确指定目标组件的Intent。它通过指定操作（Action）和数据（Data）来描述所需的操作，然后由系统根据

显示Intent适用于启动特定的组件，适用于应用内部的组件调用。

而隐式Intent适用于更通用的场景，可以调用其他应用的组件，或者通过系统提供的功能完成特定的操作，如发送邮件、拨打电话等

Fraglnet生命周期

  1. onAttach(): 当Fragment与Activity关联时调用，可以通过该方法获取到所关联的Activity的引用。
  2. onCreate(): 在Fragment创建时调用，用于初始化Fragment的状态和变量。
  3. onCreateView(): 创建Fragment的视图层次结构，并返回对应的View对象。
  4. onViewCreated(): 当Fragment的视图创建完成后调用，可以在这里进行与UI相关的操作。
  5. onActivityCreated(): 当Fragment所属的Activity的onCreate()方法执行完成时调用。
  6. onStart(): 当Fragment可见时调用。
  7. onResume(): 当Fragment获取焦点并与用户进行交互时调用。
  8. onPause(): 当Fragment失去焦点但仍然可见时调用。
  9. onStop(): 当Fragment完全不可见时调用。
  10. onDestroyView(): 当Fragment的视图被移除时调用。
  11. onDestroy(): 当Fragment被销毁时调用。
  12. onDetach(): 当Fragment与Activity解除关联时调用。

Andriod按下Home键或者返回键对Activity的影响

  1. Home 键：按下 Home 键会将当前活动（Activity）发送到后台，回到主屏幕，并将应用程序保持在后台运行状态。活动的生命周期将进入 onPause()、onStop()、onSaveInstanceState()（如果需要保存状态）和 onUserLeaveHint() 方法。
  2. 返回键：按下返回键会导致当前活动（Activity）被销毁，并返回到前一个活动或上一层级的界面。活动的生命周期将依次调用 onPause()、onStop() 和 onDestroy() 方法。

Service启动方式，生命周期，两者之间的区别

  • 启动方式

    1. 启动式启动(Started)

      通过调用 startService() 方法来启动 Service

    2. 绑定式启动(Bound)

      通过调用 bindService() 方法来启动 Service

  • 生命周期

    1. onCreate()

      在 Service 第一次创建时调用，用于初始化一次性的设置。

    2. onStartCommand()

      在每次启动 Service 时调用，用于处理启动命令。

    3. onBind()

      在绑定式启动时调用，用于返回一个 IBinder 对象，提供与绑定的组件之间的通信接口。

    4. onUnbind()

      在所有绑定的组件解绑后调用，用于清理绑定相关的资源。

    5. onDestory()

      在 Service 销毁时调用，用于释放资源和进行最终的清理工作。

  • 区别

    • 启动式启动的 Service 可以在后台长时间运行，即使启动它的组件销毁或者退出，Service 仍然可以继续运行。而绑定式启动的 Service 在所有绑定的组件解绑后会自动停止运行。
    • 启动式启动的 Service 可以被多个组件同时启动和停止，而绑定式启动的 Service 只能与一个组件进行绑定。
    • 启动式启动的 Service 通过 startService() 方法启动，而绑定式启动的 Service 通过 bindService() 方法启动。
    • 启动式启动的 Service 可以通过 Intent 传递参数，而绑定式启动的 Service 通过 IBinder 接口与绑定的组件进行通信。

SqliteopenHelper，SqliteDatabase的用法

SQLiteOpenHelper 是 Android 提供的一个帮助类，用于管理 SQLite 数据库的创建和版本管理。它是一个抽象类，需要继承并实现其中的方法。

下面是 SQLiteOpenHelper 的用法示例：

  1. 创建一个继承自 SQLiteOpenHelper 的子类：
xxxxxxxxxx
javaCopy codepublic class DBHelper extends SQLiteOpenHelper {
    private static final String DATABASE_NAME = "mydatabase.db";
    private static final int DATABASE_VERSION = 1;
​
    public DBHelper(Context context) {
        super(context, DATABASE_NAME, null, DATABASE_VERSION);
    }
​
    @Override
    public void onCreate(SQLiteDatabase db) {
        // 创建数据库表格的逻辑
        String createTableQuery = "CREATE TABLE mytable (id INTEGER PRIMARY KEY, name TEXT)";
        db.execSQL(createTableQuery);
    }
​
    @Override
    public void onUpgrade(SQLiteDatabase db, int oldVersion, int newVersion) {
        // 数据库版本升级的逻辑
        String dropTableQuery = "DROP TABLE IF EXISTS mytable";
        db.execSQL(dropTableQuery);
        onCreate(db);
    }
}
  1. 在需要使用数据库的地方，创建 DBHelper 的实例并获取 SQLiteDatabase 对象：
xxxxxxxxxx
javaCopy codeDBHelper dbHelper = new DBHelper(context);
SQLiteDatabase db = dbHelper.getWritableDatabase();

通过 getWritableDatabase() 方法获取可写的数据库对象，如果数据库不存在，则会调用 onCreate() 方法创建数据库和表格；如果数据库已存在但版本号不一致，则会调用 onUpgrade() 方法进行升级操作。

Andriod广播

  1. 标准广播（Normal Broadcast）：发送广播后，所有匹配该广播的接收器（Broadcast Receiver）几乎同时接收到广播消息，没有特定的顺序。这种广播的执行效率较高，但无法中断或修改广播。
  2. 有序广播（Ordered Broadcast）：发送广播后，接收器按照优先级依次接收广播消息，并可以中断广播或修改广播内容。每个接收器处理完广播后可以选择继续传递广播或终止广播。

使用广播的基本步骤如下：

  1. 定义广播接收器（Broadcast Receiver）：创建一个继承自 BroadcastReceiver 的子类，重写 onReceive() 方法，在该方法中处理接收到的广播消息。
xxxxxxxxxx
javaCopy codepublic class MyBroadcastReceiver extends BroadcastReceiver {
    @Override
    public void onReceive(Context context, Intent intent) {
        // 处理接收到的广播消息
    }
}
  1. 注册广播接收器：可以在 AndroidManifest.xml 文件中静态注册广播接收器，也可以在代码中动态注册广播接收器。
  • 静态注册广播接收器：

在 AndroidManifest.xml 文件中添加以下代码：

xxxxxxxxxx
xmlCopy code<receiver android:name=".MyBroadcastReceiver">
    <intent-filter>
        <action android:name="com.example.MY_ACTION" />
    </intent-filter>
</receiver>
  • 动态注册广播接收器：

在代码中使用以下方法进行动态注册：

xxxxxxxxxx
javaCopy codeMyBroadcastReceiver receiver = new MyBroadcastReceiver();
IntentFilter filter = new IntentFilter("com.example.MY_ACTION");
context.registerReceiver(receiver, filter);
  1. 发送广播：通过 Intent 对象创建广播消息，并通过 sendBroadcast() 或 sendOrderedBroadcast() 方法发送广播。
xxxxxxxxxx
javaCopy codeIntent intent = new Intent("com.example.MY_ACTION");
intent.putExtra("key", "value");
context.sendBroadcast(intent);

通过以上步骤，广播发送方发送广播消息，接收方的广播接收器会接收到该消息，并在 onReceive() 方法中处理相应的逻辑。

需要注意的是，广播接收器在不需要时应及时取消注册，以避免内存泄漏。动态注册的广播接收器应在适当的时机调用 unregisterReceiver() 方法取消注册。

上述代码中的 setConnectTimeout() 和 setReadTimeout() 方法分别用于设置连接超时时间和读取超时时间，单位是毫秒。如果在指定的时间内无法建立连接或读取数据，则会抛出 java.net.SocketTimeoutException 异常。

根据实际需求，可以根据业务逻辑调整超时时间的设置。需要注意的是，设置过长的超时时间可能会导致请求响应时间过长，影响用户体验；而设置过短的超时时间可能会导致连接不稳定或数据读取不完整。因此，建议根据具体情况进行适当的调整。

AndriodManifest.xml权限配置的标签内容属性

在 AndroidManifest.xml 文件中，使用 <uses-permission> 标签来声明应用程序需要使用的权限。下面是 <uses-permission> 标签的常见属性：

  1. android:name: 指定权限的名称，例如 android.permission.ACCESS_FINE_LOCATION 表示获取精确位置的权限。

  2. android:maxSdkVersion: 指定权限的最大 SDK 版本。可以用来限制在特定的 SDK 版本下才需要该权限。

  3. android:required: 指定权限是否是必需的。如果设置为 true，则表示应用程序需要此权限才能正常运行。默认值为 true。

列表控件及其用法

  1. RecyclerView：RecyclerView 是一种高度可定制的、灵活的列表控件，用于展示大量数据。它通过使用 ViewHolder 模式和布局管理器来提供高效的列表项的回收和重用。使用 RecyclerView，你可以自定义列表项的布局和交互，并实现各种列表效果，如垂直列表、水平列表、网格布局等。
  2. ListView：ListView 是 Android 中最常用的列表控件之一。它以垂直方式展示列表项，支持单选、多选、自定义布局等功能。你可以使用 ArrayAdapter 或自定义的适配器来填充数据，并设置点击事件或长按事件来处理用户交互。
  3. GridView：GridView 是用于展示二维网格布局的列表控件。它以行和列的方式显示数据，并支持自定义布局和适配器来填充数据。GridView 可用于展示图标、图片、网格布局等。
  4. Spinner：Spinner 是下拉选择框，常用于展示一组选项供用户选择。它可以设置自定义适配器来填充数据，并监听选中项的变化。
  5. ExpandableListView：ExpandableListView 是可展开的列表控件，用于展示带有分组和子项的数据。每个分组可以展开或折叠，显示相应的子项数据。你可以使用自定义适配器来填充分组和子项的数据，并实现相应的展开和折叠效果。

Handler 、 MeSSage 、 Looper 机制，消息发送和接收处理的方法

Handler是用于发送和处理消息的主要类，它可以将消息发送到目标线程的消息队列中，并在目标线程中处理消息。通过Handler，我们可以在不同线程之间传递消息并进行相应的处理。

Message是消息的载体，它包含了要传递的数据和标识信息。我们可以通过Handler的obtainMessage方法获取一个Message对象，并设置其属性值，然后通过Handler的sendMessage方法将消息发送到目标线程的消息队列中。

Looper是消息循环器，它用于不断地从消息队列中取出消息并进行处理。每个线程只能有一个Looper对象，它负责管理该线程的消息队列。在目标线程中，我们需要调用Looper的prepare方法创建一个Looper对象，并通过调用Looper的loop方法启动消息循环。

消息的发送和接收处理方法如下：

  1. 发送消息：我们可以通过Handler的sendMessage方法发送消息。可以通过obtainMessage方法获取一个Message对象，并设置其属性值，然后调用sendMessage将消息发送到目标线程的消息队列中。
  2. 接收消息和处理：目标线程中的Looper负责不断地从消息队列中取出消息，并将其分发给Handler进行处理。我们可以通过重写Handler的handleMessage方法来处理不同类型的消息。当消息到达目标线程时，Looper会将消息传递给Handler，并调用handleMessage方法进行处理。

在消息的处理过程中，可以根据消息的不同类型进行不同的处理逻辑。可以通过消息的what属性来区分不同类型的消息，并通过obj属性来传递消息的附加数据。

Handler线程通信机制

  1. 在目标线程中创建一个handler对象，并与looper关联
  2. 在发送线程中创建一个message对象，将需要传递的数据设置到message中
  3. 调用handler中的sendMessage()方法，将message消息发送到消息队列中
  4. 目标线程会不断从消息队列中提取消息，并交给handler处理
  5. 目标线程中handler根据接收到的消息类型进行相应的处理

 

AsynTask内部实现的方法

  1. onPreExecute(): 在执行异步任务之前调用，通常用于进行一些准备工作，例如显示进度条或初始化UI组件。
  2. doInBackground(Params... params): 在后台线程中执行异步任务的主要逻辑。该方法在onPreExecute()之后调用，用于执行耗时操作，例如网络请求或数据库操作。在该方法中可以通过publishProgress()方法发布进度更新，通知UI线程更新进度。
  3. onProgressUpdate(Progress... values): 在主线程中调用，用于更新UI的进度。该方法在调用publishProgress()后被触发，可以根据传递的进度值进行相应的UI更新操作。
  4. onPostExecute(Result result): 在主线程中调用，异步任务执行完毕后调用。该方法接收doInBackground()的返回结果作为参数，可以根据结果进行相应的UI更新或其他操作。
  5. onCancelled(): 在主线程中调用，当异步任务被取消时调用。可以在该方法中进行一些清理操作或通知用户任务取消的相关信息。

HarmontOS Ability：FA、PA

在HarmonyOS中，Ability（能力）是指应用程序提供的功能单元，用于完成特定的任务或提供特定的服务。Ability主要分为FA（Foreground Ability）和PA（Background Ability）两种类型。

  1. FA（Foreground Ability）：前台能力，代表应用程序的可见部分，提供与用户直接交互的功能。它可以包括界面、UI交互、事件处理等。FA运行在应用程序的主线程中，可以与用户进行交互，并具有良好的响应能力。常见的FA包括Activity、Slice等。
  2. PA（Background Ability）：后台能力，代表应用程序的后台执行部分，主要用于处理一些耗时操作、定时任务或与其他应用程序进行通信等。PA运行在独立的线程中，不与用户进行直接交互，可以在后台长时间运行而不影响应用程序的前台体验。常见的PA包括Service、Broadcast Receiver、Data Ability等。

通过将FA和PA相结合，HarmonyOS能够实现应用程序的多任务处理和资源优化，提供更加流畅和高效的用户体验。FA和PA之间可以通过能力连接（Ability Connect）进行通信和数据交换，实现不同能力的协同工作。
